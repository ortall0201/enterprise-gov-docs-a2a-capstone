### Why Agents Specifically:

**1. Agents Create Natural Boundaries**
- Traditional API: Tight coupling, shared responsibility
- **AI Agents via A2A:** Loose coupling, clear liability split
- Customer's agent = Customer's code = Customer's responsibility
- My agent = My code = My responsibility

**2. Agents Enable Client-Side Intelligence**
- Customer's agent can **decide** what to filter based on their policies
- Customer's agent can **retry** with different filters if needed
- Customer's agent can **audit** what was sent for compliance
- Not just dumb API calls - intelligent data handling on their side

**3. Agents Are Self-Describing (Agent Cards)**
- My service publishes capabilities at `/.well-known/agent-card.json`
- Customers discover what I offer without tight integration
- Can swap vendors by just pointing RemoteA2aAgent to new URL
- No custom integration code per vendor

**4. A2A Protocol = Industry Standard**
- Google-backed standard gives enterprises confidence
- Legal departments understand "standard protocol" better than "custom API"
- Easier to get security approval when following industry standards
- Future-proof: as A2A grows, my service becomes more valuable

**5. Agents Mirror Organizational Reality**
- Government ministry = One organization with its own agents
- My vendor service = Different organization with different agents
- A2A protocol = How organizations collaborate in real world
- Architecture matches business reality = Easier to reason about liability

### Why NOT Just an API?

An API says: "Send me data, I'll process it"
An Agent says: "I have capabilities you can use"

**The difference is who owns the data flow:**
- API: Vendor controls the flow (vendor liability)
- Agent: Customer controls the flow (customer liability)

This isn't just technical - it's **legal architecture**.

### Component Details

**Customer Side (Their Responsibility):**

1. **IntakeAgent** (`agents/intake_agent.py`)
   - Validates document exists
   - Extracts metadata (size, format, document ID)
   - First line of defense
   - Tool: `validate_document()`

2. **ProcessingAgent** (`agents/processing_agent.py`)
   - Multi-step pipeline orchestration
   - Calls OCR tool for text extraction
   - **Applies PII filtering BEFORE vendor call**
   - Delegates to RemoteA2aAgent for translation
   - Verifies vendor response (no PII leakage)
   - Tools: `ocr_tool()`, `security_filter()`
   - Sub-agents: `RemoteA2aAgent`

3. **Security Layer** (`security/policy.py`)
   - Detects 7 PII pattern types:
     - National IDs (Spanish format: `123-45-6789-X`)
     - SSN (`123-45-6789`)
     - Phone numbers (international)
     - Email addresses
     - Credit card numbers
     - Dates of birth
     - Passport numbers
   - Masking strategies:
     - Emails: `m*************@ejemplo.es` (show first char + domain)
     - IDs: `***-**-****-X` (show last 4)
     - Dates: `XX de XXXX, 1990` (mask day/month, keep year)
   - Post-vendor verification (ensures vendor didn't leak PII)

4. **RemoteA2aAgent** (`tools/vendor_connector.py`)
   - Factory function: `create_remote_vendor_agent()`
   - Points to vendor's agent card URL
   - Handles A2A protocol communication
   - Used as sub-agent in ProcessingAgent

**Vendor Side (My VaaS):**

1. **Docs Translator Agent** (`vendor/docs_translator_agent.py`)
   - My original CrewAI-based translation app
   - Wrapped in ADK LlmAgent
   - Tools:
     - `translate_document()` - ESâ†’EN translation
     - `validate_translation()` - Quality checks
   - Receives **only filtered data** (no raw PII)

2. **A2A Server** (`vendor/vendor_server.py`)
   - Exposes agent via `to_a2a(my_agent)`
   - Creates standard endpoints:
     - `/.well-known/agent-card.json` (Agent Card)
     - `/streams` (Streaming endpoint)
   - FastAPI + Uvicorn
   - Runs independently (separate process/infrastructure)

### Key Technical Decisions

- Low latency (government agents need speed)
- Cost-effective for high-volume processing

**2. Why Regex for PII Detection?**
- Fast (no LLM calls needed)
- Deterministic (same input = same output)
- Auditable (clear rules for compliance)
- Pattern library can be customized per customer

**3. Why Sub-Agent Pattern?**

- Vendor is just another capability of ProcessingAgent
- Clean delegation pattern
- Easy to swap vendors (just change RemoteA2aAgent URL)

**4. Why Separate Intake and Processing Agents?**
- Separation of concerns (validate â‰  process)
- Can add more validators without touching processing logic
- Mirrors real government workflow (intake desk â†’ processing dept)

**5. Why CrewAI for Vendor Agent?**
- My original app was already CrewAI-based
- Shows you can wrap existing apps in A2A (not just new builds)
- Demonstrates interoperability (CrewAI + ADK working together)
